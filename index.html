Project Overview

This guide provides an interactive roadmap for porting ev3dev to Debian 13 "Trixie". It synthesizes a detailed technical analysis into an actionable plan, designed for experienced developers. The core challenge is the significant version gap between ev3dev's current base (Debian 9/10) and Trixie, compounded by the inaccessibility of the old ev3dev package archive. This necessitates a modern, source-based build approach.

Primary Recommendation: A Docker-Centric Workflow

The most robust and maintainer-aligned strategy is to fully embrace a Docker-centric workflow. This involves creating a custom Debian Trixie cross-compilation environment within Docker, systematically rebuilding all ev3dev-specific components (like `ev3devKit`) from source for the `armel` architecture, and finally, using the `brickstrap` tool to assemble a bootable SD card image from a final, consolidated Docker image.

The Challenge

Porting to Trixie is not a simple package upgrade. This section breaks down the three main hurdles: the significant version gap between Debian releases, the inherent instability of a "testing" branch, and the complexities of ensuring hardware driver compatibility on the new base.

Debian Version Gap

ev3dev's stable version is based on Debian 9 (Stretch), with development on Debian 10 (Buster). Our target, Trixie, is Debian 13. This three-to-four-generation leap means major changes in core libraries (like GLib, GObject), toolchains, and kernel APIs, which will likely break existing build scripts and require code-level adaptations.

The Developer's Toolkit

A successful port relies on a specific set of tools. This section outlines the essential utilities for cross-compilation, package building, and image creation. The entire workflow is orchestrated within a Docker environment, which provides the necessary isolation and reproducibility.

Core Environment

  • Docker: For creating isolated, reproducible cross-compilation environments. The cornerstone of the modern ev3dev build process.
  • Brickstrap: The final-stage tool used to assemble a bootable `.img` file from a completed Docker image.

Package Building

  • dpkg-buildpackage: The primary script for automating the build of Debian source packages into `.deb` files.
  • crossbuild-essential-armel: A critical metapackage that installs the entire `armel` cross-compilation toolchain.
  • devscripts & dh-make: A collection of helper scripts for Debian developers, including tools to "Debianize" source code.

Package Management & QA

  • apt & dpkg: Standard Debian tools for managing packages and dependencies within the build environment.
  • lintian & piuparts: Quality assurance tools to check for policy violations and ensure packages install/remove correctly.

The Porting Roadmap

This section provides a sequential, step-by-step guide to the porting process. Each step is a major phase in the journey, from setting up your build environment to flashing a final image. Click on each step to expand it and view the detailed actions required.

The foundation of this project is a custom Docker image based on Debian Trixie, equipped with the `armel` cross-compiler. This ensures a clean, isolated, and reproducible build environment.

  1. Clone `ev3dev/docker-cross` repository: This gives you the templates for existing build environments. 
    git clone https://github.com/ev3dev/docker-cross.git
cd docker-cross
  2. Create a `debian-trixie-cross` Dockerfile: Adapt an existing file (e.g., `debian-buster-cross/Dockerfile`). Set the base image to `FROM debian:trixie` and install essential tools like `crossbuild-essential-armel`. Ensure you add the `armel` architecture with `dpkg --add-architecture armel`.
  3. Build the custom Docker image: 
    docker build -t ev3dev/debian-trixie-cross .
  4. Verify the cross-compiler: Run a container and perform a quick test compilation to confirm the `arm-linux-gnueabi-gcc` toolchain is working. 
    docker run --rm -it ev3dev/debian-trixie-cross bash
# Inside container:
arm-linux-gnueabi-gcc -o hello -x c <(echo 'int main(){return 0;}')
file hello # Should report: ELF 32-bit LSB executable, ARM...

This is the most iterative phase. You must identify all custom ev3dev packages (e.g., `ev3devKit`, `brickman`, `ev3dev-tools`), update their Debian packaging files to be Trixie-compatible, and rebuild them from source.

  1. Obtain Source Code: Clone the GitHub repository for each custom ev3dev component (e.g., `git clone https://github.com/ev3dev/ev3devKit.git`).
  2. Update Dependencies: In the `debian/control` file of each package, update the `Build-Depends` to match library versions available in Trixie. This will require research and testing.
  3. Adapt Build Rules: Modify the `debian/rules` file if necessary to work with Trixie's newer compilers and build systems.
  4. Cross-Compile the Package: Use `dpkg-buildpackage` inside your Trixie cross-compilation container to build the `.deb` file for the `armel` architecture. 
    # Mount the source directory and run the build
docker run --rm -it -v $(pwd)/ev3devKit:/src -w /src ev3dev/debian-trixie-cross \
  dpkg-buildpackage -us -uc -b -aarmel
  5. Iterate and Debug: Expect compilation failures. Read error messages carefully—they often point to missing `-dev` packages or API incompatibilities that need to be fixed in the source code or packaging files. Repeat until all components build successfully.

With all custom components rebuilt as `.deb` packages, the next step is to assemble them into a final, comprehensive Docker image that represents the complete ev3dev-Trixie operating system.

  1. Create a Final Dockerfile: Adapt a Dockerfile from the `ev3dev/docker-library` repository. Set the base to `FROM debian:trixie`.
  2. Install Custom Packages: Add `COPY` and `RUN dpkg -i` commands to your Dockerfile to install all the custom `.deb` packages you built in the previous step.
  3. Build the Final Image: 
    docker build -t ev3dev/ev3dev-trixie-ev3-generic .

The final step is to convert your complete Docker image into a bootable SD card image using `brickstrap`, flash it, and perform rigorous testing on the actual EV3 hardware.

  1. Generate Image with `brickstrap`: Use the `brickstrap` tool to convert your final Docker image into a bootable `.img` file. 
    brickstrap create-tar ev3dev/ev3dev-trixie-ev3-generic ev3dev-trixie.tar
brickstrap create-image ev3dev-trixie.tar ev3dev-trixie.img
  2. Flash the Image: Use a tool like Balena Etcher or `dd` to write the `ev3dev-trixie.img` file to a microSD card.
  3. Comprehensive Hardware Testing: Boot the EV3 from the new SD card. Systematically test every feature: motors, all sensor ports, display, buttons, sound, USB, and network connectivity. Document any bugs or regressions for further debugging.

Challenges & Key Risks

This project, while feasible, carries inherent risks due to the nature of the software involved. Understanding these challenges upfront is crucial for managing expectations and planning for long-term maintenance.

Instability of a "Testing" Release

Debian Trixie is a moving target. As the "testing" branch, it receives continuous updates, and there is no guarantee of API stability. A system built on Trixie today might break after a routine `apt upgrade` tomorrow. This requires a commitment to ongoing maintenance and debugging until Trixie becomes stable.

Hardware Driver Compatibility

ev3dev's functionality depends on custom kernel modules and drivers. These must be compatible with Trixie's newer Linux kernel. This may require deep, technical work, including patching driver source code to align with new kernel APIs, which can be the most time-consuming part of the port.

Final Recommendations

To maximize the chances of a successful port and create a sustainable system, follow these best practices throughout the development process. A disciplined, systematic approach is key to navigating the complexities of this project.

  • Prioritize and verify the `armel` cross-compilation toolchain at every step.
  • Leverage existing `ev3dev/docker-cross` and `ev3dev/docker-library` Dockerfiles as templates.
  • Prepare for an iterative cycle of building, testing, and debugging, especially for custom packages.
  • Rigorously test all hardware functions on the EV3 brick after creating a bootable image.
  • Proactively map dependencies before compiling and be ready for manual conflict resolution.
  • Engage with the ev3dev and Debian communities for support and troubleshooting.
  • Plan for long-term maintenance, as Trixie will evolve until its official stable release.
  • Document every change, dependency resolution, and patch for future reference.
